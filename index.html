transfer

Simple, but powerful rsync based backup mechanism.

Download as .zip Download as .tar.gz View on GitHub

What is this about?

This project is a simple incremental backup script that uses rsync to transfer files.

Here is an example of how you might call it once:

transfer -r example.net -s apache <<EOF
/var/www/
/srv/www/
/etc/apache2/
EOF

And what it may give you in return:

/var/backups/hosts.d/ # the base directory to store backups into
└── example.net       # remote
     └── apache       # subject
         ├── empty    
         ├── err      # failed copies
         ├── ok       # copies that finished correctly
         │   ├── 20120405195300  # date/time
         │   │   ├── data.d      # actuall files
         │   │   ├── start       # start date/time
         │   │   ├── end         # finish date/time
         │   │   ├── subject     # a name like apache or files
         │   │   ├── remote      # remote server hostname
         │   │   ├── local       # backup server hostname
         │   │   ├── rsync.log   # transfer details
         │   │   ├── size        # size of files inside data.d
         │   │   ├── status      # status of this copy (ok or err)
         │   │   ├── uuid        # id
         │   │   ├── previous    # parent id
         │   │   └── version     # version of the script
         │   └── last -> /var/backups/.../ok/20120405195300
         └── tmp      # meanwhile a copy is made

Why should I use it?

Do you need to retain and query information about how a backup was made but will prefer to not use a database?

Maybe you will reallocate copies to other servers but wish you could known where did it come from?

Do not know where to place the files you want to restore again?

Want to save space but wish you don't need to wait for hours for a reconstruction process when you need some old files?

Must you store things in the filesystem directly for performance, storage restrictions or easy of access?

Absolutely need data stored in backups to be directly explorable and manipulable because you don't know where that file may be?

You wish to construct you own backup policy, create you own reports or make tools around the process?

Hate push backups because you don't want your backups to get deleted by intruders?

transfer is pretty dumb, but can help with a lot of complicated problems, and it's just around 200 lines of shellscript (maybe half of that are comments!).

Please try, and come to love or hate it.
